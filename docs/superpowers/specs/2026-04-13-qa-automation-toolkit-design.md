# QA Automation Toolkit — Design Spec

**Date:** 2026-04-13
**Status:** Draft
**Prototype repo:** storium (project-level, `.claude/` directory)
**Target repo:** minga monorepo (future — convert to plugin)

---

## Problem

The QA team is shrinking while AI-assisted development is accelerating code output. The codebase is a monorepo with uneven test coverage: some apps have unit tests, newer systems have integration tests with testcontainers, and a separate Playwright app runs e2e tests. There is no unified view of what is covered, what is fragile, or where to focus QA effort. The code review process is bottlenecking releases.

## Goals

1. **Aggregated coverage report** — unified view across unit, integration, and e2e tests
2. **Fragility analysis** — identify high-risk, low-coverage, critical-domain files
3. **Test-to-file tracing** — given a test, show what it covers; given a file, show what tests cover it
4. **Auto-generate missing tests** — LLM-powered unit test generation for uncovered files
5. **Test ID registry** — unique IDs (QA-10000+) for every test, tracked with metadata
6. **Accumulated learnings** — cross-run knowledge that makes each execution smarter

## Non-Goals

- Replacing existing test infrastructure (vitest, jest, playwright)
- Running tests in CI (this is an on-demand analysis tool)
- Modifying production code (only test files and `.minga/qa/` data)

## Architecture

### Approach: Skills backed by TypeScript CLI

A small TypeScript module in `.claude/tools/qa/` handles all deterministic work: running external tools, parsing their output, merging metrics, computing scores. Claude Code skills in `.claude/skills/qa-*/SKILL.md` orchestrate the pipeline and provide LLM-powered analysis on top.

This separation means:
- **Deterministic data** (metrics, scores, coverage) is reproducible and auditable
- **LLM judgment** (recommendations, test generation, learnings) is applied where it adds value
- **Skills stay thin** — invoke the CLI tools via `!` shell injection, then analyze the structured output

### Generated Markdown Files

All `.md` files in `.minga/qa/` are auto-generated and not human-maintained. They are regenerated by skills whenever their upstream JSON data changes. They serve as human-readable views of the structured data.

| File | Purpose | Style |
|------|---------|-------|
| `catalog.md` | Overview of all tracked files with scores, domains, classification | Tabular, sortable-feeling |
| `coverage.md` | Aggregated coverage by domain/module, uncovered critical areas | Summary stats + breakdown |
| `analysis.md` | Full QA analysis: trends, regressions, health assessment | Narrative with data |
| `recommendations.md` | Action items only — what to fix next, prioritized | Checklist / action items |
| `learnings.md` | Dashboard of accumulated learnings grouped by domain | Status dashboard, scannable |

### Regeneration Rules

| Skill | JSON mutated | Markdown regenerated |
|-------|-------------|---------------------|
| `/qa-analyze` | catalog.json, history/ | All five |
| `/qa-coverage` | coverage.json, catalog.json (coverage fields) | coverage.md, analysis.md, recommendations.md, learnings.md (if learnings changed) |
| `/qa-fragile` | None (read-only) | analysis.md, recommendations.md, learnings.md (if learnings changed) |
| `/qa-trace` | None (read-only) | learnings.md (if learnings changed) |
| `/qa-make-tests` | test-registry.json, coverage.json | coverage.md, recommendations.md, learnings.md (if learnings changed) |
| `/qa-test-id` | test-registry.json | catalog.md, learnings.md (if learnings changed) |

## Data Models

### catalog.json — Per-File Entry

```typescript
type CatalogEntry = {
  path: string                    // relative file path

  // Classification (inferred)
  classification: 'source' | 'test' | 'config' | 'script' | 'fixture' | 'docs'
  domain: string | null           // inferred: 'auth', 'payments', 'core', etc.
  criticalityScore: number        // 0-100, from fan-in + export count + domain weight

  // Fallow metrics (deterministic)
  fallow: {
    maintainabilityIndex: number
    complexityDensity: number
    totalCyclomatic: number
    totalCognitive: number
    fanIn: number
    fanOut: number
    deadCodeRatio: number
    crapMax: number
    lines: number
    hotspot?: {
      score: number
      commits: number
      churn: number
      trend: 'heating' | 'cooling' | 'stable'
    }
  }

  // Git metrics (deterministic)
  git: {
    commits6mo: number
    authors6mo: number
    lastModified: string          // ISO date
    age: string                   // ISO date of first commit
  }

  // TypeScript health (deterministic)
  tsc: {
    errorCount: number
    errors: string[]              // error codes: ['TS2345', 'TS7006']
  }

  // Dependency graph (deterministic)
  deps: {
    dependsOn: string[]           // files this file imports
    dependedOnBy: string[]        // files that import this file
    circularWith: string[]        // circular dependency partners
    depth: number                 // max depth in dependency tree
  }

  // Test coverage (deterministic, from vitest/istanbul)
  coverage: {
    lineCoverage: number          // 0-100 percentage
    branchCoverage: number
    functionCoverage: number
    coveredByTests: string[]      // test IDs: ['QA-10001', 'QA-10003']
    lastCoverageRun: string       // ISO date
  }

  // Composite scores (computed by score.ts)
  fragilityScore: number          // 0-100, higher = more fragile
  qualityScore: number            // 0-100, higher = better quality
  priorityRank: number            // 1-N, overall "fix me first" ranking

  lastUpdated: string             // ISO date
}
```

### test-registry.json — Per-Test Entry

Test IDs start at QA-10000 and increment sequentially.

```typescript
type TestRegistryEntry = {
  id: string                      // QA-10000, QA-10001, ...
  name: string                    // test description string
  suite: string                   // describe block or file-level suite name
  file: string                    // path to test file
  line: number                    // line number in file
  type: 'unit' | 'integration' | 'e2e'
  status: 'active' | 'disabled' | 'missing' | 'stale'
  coveredFiles: string[]          // source files this test exercises
  coveredFunctions: string[]      // specific functions exercised (when detectable)
  lastVerified: string            // ISO date
  lastPassed: string | null       // ISO date
  createdAt: string               // ISO date
}
```

### Test ID Convention

Test IDs appear in both code comments (for scanning) and test names (for runner output):

```typescript
/* QA-10001 */ it('[QA-10001] should hash password before storing', async () => {
  // ...
})
```

- `/* QA-NNNNN */` — block comment for grep/scanner to find and parse
- `[QA-NNNNN]` — in test name so vitest/jest output shows the ID
- Humans and agents can correlate test runner output to registry entries

### learnings.json

```typescript
type Learning = {
  id: string                       // short hash for dedup/reference
  domain: string                   // 'coverage' | 'fragility' | 'testing' | 'architecture' | 'tooling'
  insight: string                  // the actual knowledge
  confidence: 'low' | 'medium' | 'high'
  source: string                   // which command: 'qa-analyze', 'qa-make-tests', etc.
  firstSeen: string                // ISO date
  lastConfirmed: string            // ISO date
  supersedes?: string              // id of learning this one replaced/refined
}

type LearningsFile = {
  version: 1
  entries: Learning[]
}
```

**Merge rules (LLM-enforced in skill instructions):**

| Situation | Action |
|-----------|--------|
| New insight, nothing similar exists | Append with `confidence: 'low'` |
| New insight confirms an existing one | Update `lastConfirmed`, bump confidence |
| New insight contradicts an existing one | Add new entry with `supersedes` pointing to old |
| New insight refines/expands existing | Synthesize into existing entry's `insight` text |
| Existing insight not re-observed in 5+ runs | No action (stale is fine, still true) |

### history/ — Timestamped Snapshots

```typescript
type HistorySnapshot = {
  timestamp: string                // ISO datetime
  command: string                  // which skill triggered this

  // What changed in each generated markdown file
  diffs: {
    file: string                   // 'catalog.md' | 'coverage.md' | 'analysis.md' | 'recommendations.md' | 'learnings.md'
    removed: string[]              // sections/items removed or replaced (summarized)
    added: string[]                // sections/items added
    summary: string                // one-line description of what changed and why
  }[]

  // Key metrics at this point in time (for trending)
  metrics: {
    totalFiles: number
    avgFragilityScore: number
    avgQualityScore: number
    totalCoverage: number          // aggregate line coverage %
    testsTracked: number
    criticalUncovered: number      // criticalityScore > 70 and lineCoverage < 50
  }
}
```

Before any generated markdown file is overwritten, the skill reads the old content, diffs against the new, summarizes what was removed/replaced, and records it in the history entry. This means you can trace why a recommendation disappeared or a score changed.

## Tool Pipeline

### collect.ts — Deterministic Data Collection

Runs external tools in parallel where possible, collects JSON output:

```
+-----------+  +-------------------+  +-----------+  +---------+
| fallow    |  | dependency-cruiser|  | tsc       |  | git log |
| --format  |  | --output-type     |  | --noEmit  |  | per-file|
| json      |  | json              |  | (parse)   |  | churn   |
+-----------+  +-------------------+  +-----------+  +---------+
      |                |                    |              |
      +----------------+--------------------+--------------+
                              |
                         collect.ts
                    (merge into raw per-file records)
```

**External tools used:**
- `fallow --format json --explain` — maintainability, complexity, churn hotspots, fan-in/out, CRAP, dead code
- `dependency-cruiser --output-type json` — full dependency graph, circular deps, depth
- `tsc --noEmit` — type error count per file (parsed from output)
- `git log` — commits per file (6mo), authors per file (6mo), last modified, file age

**Coverage collection** is separate (triggered by `/qa-coverage`):
- `vitest run --coverage --reporter=json` or `jest --coverage --json`
- Parse istanbul JSON for per-file line/branch/function coverage
- Map covered files back to test files

### score.ts — Composite Scoring

**Fragility Score (0-100)** — higher = more fragile, needs attention:

| Signal | Weight | Source | Logic |
|--------|--------|--------|-------|
| Inverse coverage | 25% | vitest/istanbul | `(100 - lineCoverage)` |
| Complexity density | 20% | fallow | Normalize to 0-100 |
| Churn intensity | 15% | fallow hotspot + git | Recent commits x churn magnitude |
| Fan-in (blast radius) | 15% | fallow + dep-cruiser | Normalized: high fan-in = fragile |
| CRAP score | 10% | fallow | Complexity adjusted for coverage |
| Stale test ratio | 10% | test registry | Tests that are `stale` or `disabled` |
| Type error density | 5% | tsc | Errors per 100 LOC |

**Criticality Score (0-100)** — how important is this file:

| Signal | Weight | Logic |
|--------|--------|-------|
| Fan-in | 35% | More dependents = more critical |
| Domain weight | 30% | Directory heuristics (see below) |
| Export count | 20% | More public surface = more critical |
| Dependency depth | 15% | Deep in graph = foundational |

**Quality Score (0-100)** — higher = better:
- Base: maintainability index (from fallow, already 0-100)
- Penalties: dead code ratio, circular deps, type errors, low coverage

**Priority Rank** — all source files sorted by `fragilityScore * criticalityScore / qualityScore`, descending. #1 is the most fragile, most critical, lowest quality file.

### Domain Inference

```typescript
const DOMAIN_WEIGHTS: Record<string, number> = {
  // High criticality (80-100)
  auth: 95, authentication: 95, payments: 95, billing: 90,
  checkout: 90, security: 90, crypto: 85, session: 85,

  // Medium-high (60-79)
  users: 75, accounts: 75, api: 70, core: 70,
  database: 70, db: 70, middleware: 65, routes: 65,

  // Medium (40-59)
  services: 55, models: 55, controllers: 50,
  hooks: 50, stores: 50, state: 50,

  // Low (0-39)
  utils: 30, helpers: 30, lib: 35, shared: 35,
  config: 25, constants: 20, types: 20,

  // Minimal
  test: 5, tests: 5, fixtures: 5, mocks: 5,
  scripts: 10, docs: 5, examples: 5,
}
```

Domain is inferred from the first 2-3 path segments. Fan-in and export count can override — a `utils/hash.ts` with 40 fan-in gets boosted above its directory weight.

## Skills

All skills are `disable-model-invocation: true` — QA team actions, not auto-triggered.

### /qa-analyze — Full Pipeline

**Arguments:** `[--coverage] [--force]`

1. Run `collect.ts` — produces raw metrics from fallow, dep-cruiser, tsc, git
2. Run `score.ts` — computes composite scores
3. Run `catalog.ts` — merges into catalog.json, saves history snapshot
4. Read catalog.json, compare with previous snapshot if exists
5. LLM generates all five markdown files
6. LLM updates learnings.json (synthesize, don't just append)
7. Print summary to terminal

### /qa-coverage — Coverage Report

**Arguments:** `[--runner vitest|jest] [--file <path>]`

1. Run vitest/jest with coverage (detect runner: check for vitest.config.* first, then jest.config.*, then package.json scripts; error with guidance if neither found)
2. Run `coverage.ts` — parse istanbul JSON output, update catalog coverage fields
3. Update coverage.json and catalog.json
4. LLM regenerates coverage.md, analysis.md, recommendations.md
5. LLM updates learnings.json if new insights
6. Regenerate learnings.md if learnings changed
7. Print aggregated coverage report to terminal

### /qa-fragile — Fragile Areas

**Arguments:** `[--top N] [--domain <name>]`

1. Read catalog.json (error if missing — tell user to run `/qa-analyze` first)
2. Sort by `fragilityScore * criticalityScore`, take top N (default 20)
3. LLM generates narrative for each: why fragile, what the risk is, what to do
4. LLM regenerates analysis.md, recommendations.md
5. LLM updates learnings.json if new insights
6. Regenerate learnings.md if learnings changed
7. Print to terminal

### /qa-trace — Test/File Mapping

**Arguments:** `<test-id-or-file-path>`

1. Read catalog.json and test-registry.json
2. If argument is a test file or test ID (QA-NNNNN) -> look up `coveredFiles` from registry
3. If argument is a source file -> look up `coveredByTests` from catalog
4. LLM presents the dependency chain: what tests exercise this code, through what paths, gaps
5. LLM updates learnings.json if new insights
6. Regenerate learnings.md if learnings changed
7. Print to terminal (no markdown file changes)

### /qa-make-tests — Generate Missing Tests

**Arguments:** `<file-path>`

1. Read the target file and its catalog entry
2. Identify uncovered functions (from coverage data + LLM code analysis)
3. LLM generates unit tests following project conventions (vitest patterns, existing test style)
4. Auto-assign test IDs from registry (next available after QA-10000)
5. Write test file with `/* QA-NNNNN */ it('[QA-NNNNN] ...` convention
6. Update test-registry.json
7. LLM regenerates coverage.md, recommendations.md
8. LLM updates learnings.json if new insights
9. Regenerate learnings.md if learnings changed

### /qa-test-id — Test ID Management

**Arguments:** `assign | lookup <QA-NNNNN> | verify`

- **assign**: Glob all test files, parse `it()`/`test()` blocks, find any without a `/* QA-NNNNN */` annotation, assign next sequential ID, add comment + name prefix, update registry
- **lookup QA-10042**: Print full metadata for that test ID from registry
- **verify**: Walk the registry, check each test still exists at its recorded file:line, update `status` field (`active`, `disabled`, `missing`, `stale`), report findings
- Update test-registry.json
- LLM regenerates catalog.md (test counts per file)
- LLM updates learnings.json if new insights
- Regenerate learnings.md if learnings changed

## Future: Plugin Conversion

When moving to the minga monorepo, convert from project-level to plugin:

```
minga-qa/
├── .claude-plugin/
│   └── plugin.json               # name: "qa", version, description
├── skills/
│   ├── analyze/SKILL.md          # becomes /qa:analyze
│   ├── coverage/SKILL.md         # becomes /qa:coverage
│   ├── fragile/SKILL.md          # becomes /qa:fragile
│   ├── trace/SKILL.md            # becomes /qa:trace
│   ├── make-tests/SKILL.md       # becomes /qa:make-tests
│   └── test-id/SKILL.md          # becomes /qa:test-id
├── bin/
│   └── qa-tools/                 # TypeScript CLI tools (added to PATH)
└── README.md
```

The plugin namespace `qa` gives the colon-separated names (`/qa:analyze`). The `.claude/tools/qa/` scripts move to `bin/qa-tools/` and are automatically on PATH when the plugin is enabled.
